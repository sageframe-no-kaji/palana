# ho-10.1 — The ZFS Tool

The reason pālana exists that no other Mac file manager does: composing `zfs` in a GUI, and enacting it through the same trust gate as every other mutation. ho-10 built the Workbench boundary and proved it read-only—`df`, `zfs list`, `zpool status`, `zpool list` run raw through the Conduit. It named the mutation seam and left it refused: a `WorkbenchTool` declares a `.mutation` verb, the coordinator is meant to route it to a `PlanRequest` instead of running it, and `Workbench.run` throws `notARead` if a mutation reaches the read path. This ho fills that seam with the first mutating tool—dataset create, destroy, rename; snapshot create, destroy, rollback; mountpoint set and clear—each one composing a `zfs` command the operator reads in the plan panel before a single keystroke enacts it. The engine stays the one engine; the trust gate stays the one gate. His delight, recorded at ho-08: "I LOVE that it will choose the right tool." This is that, turned on ZFS itself.

**Out of scope:** pool creation and destruction—pools are physical, rare, and dangerous in a way datasets are not; the tool visualizes pools (the read verbs already do, and `zpool status` drives remain a deferred fact) but never creates or destroys them. Clone (`zfs clone`)—a create-from-snapshot, a clean v-next addition, not v1. Property editing beyond mountpoint (quota, compression, recordsize)—the mountpoint is the one property the field already reasons about (ho-9.3's `◆`); the rest wait. Sending and receiving—that is ho-06.2's transfer path, already built; this tool manages datasets in place, it does not move them between hosts. Bookmarks, holds, encryption keys.

**Resolves deferred decisions:** the ho-10 close named the split—"the mutating ZFS tool the overview named (dataset CRUD, snapshots, mountpoint set/clear) splits forward to ho-10.1"—and the ho-9.3 seal placed mountpoint management here ("mountpoint MANAGEMENT (set/clear) stays the Workbench ZFS plugin"). Both land now.

## Phase 1 — Think

### Decision 1 — A ZFS mutation is a typed verb carried on a `PlanRequest`

The seam requires it: `WorkbenchTool.planRequest(for:on:)` returns a `PlanRequest?`, so a ZFS mutation must be expressible as one. The engine grows a small, closed vocabulary rather than shoehorning ZFS into the file verbs:

```swift
public enum ZFSMutation: Sendable, Equatable, Codable {
    case createDataset(name: String, mountpoint: String?)
    case destroyDataset(name: String, recursive: Bool)
    case renameDataset(from: String, to: String)
    case snapshot(dataset: String, name: String, recursive: Bool)
    case destroySnapshot(dataset: String, name: String)
    case rollback(dataset: String, name: String)
    case setMountpoint(dataset: String, path: String)
    case clearMountpoint(dataset: String)  // zfs inherit mountpoint
}
```

`PlanOperation` gains `.zfs`; `PlanRequest` gains `zfs: ZFSMutation?` (nil for every file operation, so all existing call sites stand). The tool builds a `PlanRequest(operation: .zfs, source: <the host>, zfs: <the mutation>)`. One operation, one payload—not eight new `PlanOperation` cases polluting the file vocabulary.

### Decision 2 — One classification, `.zfsMutation`, transport `.local`; the composer switches on the payload

All ZFS mutations share a handling category: a single host (the tool's host—no transfer), a delegated `zfs` command, and a `.verify` step that reads the resulting state. So `Classification` gains exactly one case, `.zfsMutation`, and `classify()` routes `.zfs` to it. `transport()` returns `.local`. `compose()` dispatches to a `composeZFSMutation` that switches on `request.zfs` and builds the steps. This mirrors ho-9.1's template (operation → classification → compose → verify) without exploding the classification enum.

### Decision 3 — The composes are delegated, portable, and verified by reading state

Following ho-06.2's discipline—delegated permissions, no `sudo` in composed commands, property names delegated alongside verbs. Every mutation is a `zfs` command plus a `.verify` step that reads the post-state, ShellQuoted throughout:

- create: `zfs create <name>` (with `-o mountpoint=<path>` when supplied) → verify `zfs list -H -o name <name>`
- destroy dataset: `zfs destroy <name>` or `zfs destroy -r <name>` when recursive → verify the name is gone (`! zfs list <name>`)
- rename: `zfs rename <old> <new>` → verify new present, old absent
- snapshot: `zfs snapshot <dataset>@<name>` (`-r` recursive) → verify the snapshot lists
- destroy snapshot: `zfs destroy <dataset>@<name>` → verify absent
- rollback: `zfs rollback <dataset>@<name>` → verify (the snapshot still exists, is now current)
- set mountpoint: `zfs set mountpoint=<path> <dataset>` → verify `zfs get -H -o value mountpoint <dataset>`
- clear mountpoint: `zfs inherit mountpoint <dataset>` → verify

`PlanStep.Role` reuses `.create`, `.delete`, `.snapshot`, `.verify` where they fit and gains what it must (`.rollback`, `.property`)—a small honest addition, decided in AT-01 against the role vocabulary as it stands.

### Decision 4 — Destroy and rollback are safe because the plan is read before Enter

The mutation seam is not a bypass of the trust gate—it is the trust gate applied to ZFS. A `zfs destroy -r` never runs on a keystroke. The composed command, recursive flag and all, renders in the plan panel exactly as it will execute; the operator reads it; Enter enacts, nothing else does. This is ho-08's law (nothing mutates on a readable plan) turned on the most destructive verbs in the tool. Recursive destroy states its recursion in the command the operator reads. No hidden `-f`, no silent recursion.

### Decision 5 — A verb targets the dataset the operator is standing in; parameters gather before the plan

The Workbench aims at the focused pane's host (ho-10). The dataset a mutation targets is the one containing the focused pane's current directory—`ZFSTopology.datasetContaining(pane.path)`—the "operate on where I'm standing" model that matches pālana's whole posture. Create makes a child of it; snapshot snapshots it; set-mountpoint sets its mountpoint; destroy and rename act on it. The mutations that need a name or path (create's child name, snapshot's name, rename's new name, set-mountpoint's path) gather it first through a small field on the ho-9.1 naming-field precedent, then compose the `PlanRequest`, then render the plan, then Enter. The exact trigger surface—chips on the strip, a submenu, a gathered field's shape—is the hands session's feel to settle; the engine and the targeting model are decided here.

### Decision 6 — Capability-gated, fixture-only, never a real pool in the build

Every verb requires `.zfs` (the `zfsTopology` fact present), disabled with a plain reason otherwise—the ho-10 gating, unchanged. **The hard rule governs: no mutating operation runs against a real pool during the build.** Unit tests assert compose exact-strings; integration tests run the real round trip—create → snapshot → rollback → set-mountpoint → destroy—against the file-backed throwaway pool in the Lima VM (`palana-zfs`, pool `palana`), gated on `ZFSFixture.available`, `.serialized`, following `ZfsTransportsIntegrationTests` exactly. The operator's real pools become targets only when the operator drives the shipped app.
