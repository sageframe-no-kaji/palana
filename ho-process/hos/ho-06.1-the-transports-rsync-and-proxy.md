# ho-06.1 — The Transports: rsync, the proxy pipeline, progress, gates

Enactment, first half. Executes an approved Plan exactly as composed—no improvisation between approval and execution. The event stream the plan panel will echo, rsync host-to-host as the fast path, the tar-stream proxy pipeline through the operator's machine, progress parsed live, counts verified, and gated steps released only by verification. The zfs transport is ho-06.2's—it needs the pool VM, this half needs only the sshd fixture.

**Out of scope:** `zfs send/receive` (ho-06.2). Any queue—one enactment at a time. Rollback—a failed transfer's partial state is visible truth, named, not silently cleaned.

**Resolves deferred decisions** (from the ho-overview):

- Progress parsing specifics—rsync progress2 field stability (deferred decision 5, rsync half)

**Carries from ho-05:** the rsync flag set (`-a -s --info=progress2`) is committed and progress2 is the parse target. The tar proxy enacts in-process—two Conduit-configured ssh halves piped, pālana counting the bytes, since tar offers no progress2. Gates are declared in the Plan; this ho enforces them.

## Phase 1 — Think

### Decision 1 — Enactment is an event stream, and the echo is the product

`enact(plan)` returns an async throwing stream of `EnactmentEvent`: `stepBegan`, `outputChunk` (stdout/stderr bytes, live—the panel's echo), `progress`, `stepEnded`, `verified`, `finished`. The system design's claim—"these are the commands" checkable by watching them run—is this stream. The Transports run the Plan's exact command strings; the events carry what actually happened.

### Decision 2 — Two runners: the Conduit for host steps, an in-process pipeline for operator-machine steps

Steps marked `runsOn: .host` run through the injected Conduit, sessions multiplexed as ho-02 built. Steps marked `runsOn: .operatorMachine`—the proxy pipelines—are enacted in-process: pālana spawns both ssh halves itself (same `SSHConfiguration`, same ControlMaster options), pipes A's stdout into B's stdin with a byte counter in the middle, and lets pipe backpressure regulate flow. The Plan's command string remains the paste-able shell equivalent; the enactment is semantically identical with the counter as a bonus the shell version doesn't have. Both halves' stderr streams surface as echo events.

### Decision 3 — Progress: a progress2 parser for rsync, the byte counter for tar

rsync's `--info=progress2` emits carriage-return-separated updates on stdout—bytes, percent, rate. The parser works over chunked bytes, splits on `\r`/`\n`, and emits `ProgressReport(bytesTransferred, fraction, rawLine)`. Field stability hardens against the fixture's real rsync during execution. The tar pipeline reports the counter's bytes; fraction computes from `Plan.totalSize` only when the selection holds no directories—inode sizes lie about trees, and an indeterminate bar is more honest than a wrong one.

### Decision 4 — Verification composes here, runs visibly, and releases the gates

After a transfer step exits clean, the Transports compose count commands—`find` over the source entries and their transplanted names at the destination, counts compared—run them through the Conduit, and emit the result as a `verified` event. Match releases the gated steps in Plan order. Mismatch throws `EnactmentError.verificationFailed` and the gated steps never run—a move that cannot prove its copy landed does not delete anything. The verify commands appear in the echo like every other command: the gate's evidence is watchable.

### Decision 5 — Failure halts, gates stay closed, partial state is named

Any step exiting nonzero throws `EnactmentError.stepFailed(index, exitStatus, stderr tail)` and ends the stream. Gated steps are unreleased by construction. No rollback in v1: rsync re-runs converge, tar partials are visible files the operator can see and act on. The error names the step so the panel can point at the exact command that failed.

### Decision 6 — The fixture learns rsync and how to reach itself

The sshd container carries no rsync and there is one of it. Fixture surgery: `start` installs rsync in the container and provisions a self-referential ssh identity—the fixture key and an in-container `~/.ssh/config` alias (`fixture-self` → localhost:2222)—so a "cross-host" rsync from the container to `fixture-self` exercises the real command path, distinct directories standing in for distinct hosts. The tar proxy pipeline needs no surgery: its halves run from the operator's machine (the test process) toward the container, which is the real shape.

### Discovery (deferred to execution) — progress2's exact fields and the pipeline's backpressure

The parser hardens against the fixture's rsync version live. The pipeline's write path must respect the ho-01 lesson—no blocking read starvation—while letting pipe backpressure do its job; the exact FileHandle discipline lands in execution against real streams.

## Phase 2 — Execute

One bounded conversation—no agent-task decomposition. Model: `claude-fable-5`.

Order of work:

1. `EnactmentEvent`, `EnactmentError`, the `Transports` skeleton—step sequencing and gate logic unit-tested over RecordedConduit.
2. The progress2 parser with a synthetic battery, hardened live and recorded.
3. The in-process pipeline: spawn, pipe, count, backpressure—unit-tested against local processes before any fixture.
4. Fixture surgery, then live integration: an rsync-direct move enacted end to end (events, progress, verification, gated delete observed), the tar proxy likewise, and a verification-mismatch case proving gates hold.
5. Full verification rhythm; floor holds; CI green.

### Done means

- A cross-host move Plan enacts end to end against the fixture: events stream, progress moves, counts verify, and the gated delete runs only after verification.
- A failed or unverified transfer leaves gated steps unrun, typed.
- The echo stream carries every command and its real output, live.
