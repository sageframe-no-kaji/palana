# ho-06.1 ‚Äî The Transports: rsync, the proxy pipeline, progress, gates

Enactment, first half. Executes an approved Plan exactly as composed‚Äîno improvisation between approval and execution. The event stream the plan panel will echo, rsync host-to-host as the fast path, the tar-stream proxy pipeline through the operator's machine, progress parsed live, counts verified, and gated steps released only by verification. The zfs transport is ho-06.2's‚Äîit needs the pool VM, this half needs only the sshd fixture.

**Out of scope:** `zfs send/receive` (ho-06.2). Any queue‚Äîone enactment at a time. Rollback‚Äîa failed transfer's partial state is visible truth, named, not silently cleaned.

## Phase 1 ‚Äî Think

### Decision 1 ‚Äî Enactment is an event stream, and the echo is the product

`enact(plan)` returns an async throwing stream of `EnactmentEvent`: `stepBegan`, `outputChunk` (stdout/stderr bytes, live‚Äîthe panel's echo), `progress`, `stepEnded`, `verified`, `finished`. The system design's claim‚Äî"these are the commands" checkable by watching them run‚Äîis this stream. The Transports run the Plan's exact command strings; the events carry what actually happened.

### Decision 2 ‚Äî Two runners: the Conduit for host steps, an in-process pipeline for operator-machine steps

Steps marked `runsOn: .host` run through the injected Conduit, sessions multiplexed as ho-02 built. Steps marked `runsOn: .operatorMachine`‚Äîthe proxy pipelines‚Äîare enacted in-process: pƒÅlana spawns both ssh halves itself (same `SSHConfiguration`, same ControlMaster options), pipes A's stdout into B's stdin with a byte counter in the middle, and lets pipe backpressure regulate flow. The Plan's command string remains the paste-able shell equivalent; the enactment is semantically identical with the counter as a bonus the shell version doesn't have. Both halves' stderr streams surface as echo events.

### Decision 3 ‚Äî Progress: a progress2 parser for rsync, the byte counter for tar

rsync's `--info=progress2` emits carriage-return-separated updates on stdout‚Äîbytes, percent, rate. The parser works over chunked bytes, splits on `\r`/`\n`, and emits `ProgressReport(bytesTransferred, fraction, rawLine)`. Field stability hardens against the fixture's real rsync during execution. The tar pipeline reports the counter's bytes; fraction computes from `Plan.totalSize` only when the selection holds no directories‚Äîinode sizes lie about trees, and an indeterminate bar is more honest than a wrong one.

### Decision 4 ‚Äî Verification composes here, runs visibly, and releases the gates

After a transfer step exits clean, the Transports compose count commands‚Äî`find` over the source entries and their transplanted names at the destination, counts compared‚Äîrun them through the Conduit, and emit the result as a `verified` event. Match releases the gated steps in Plan order. Mismatch throws `EnactmentError.verificationFailed` and the gated steps never run‚Äîa move that cannot prove its copy landed does not delete anything. The verify commands appear in the echo like every other command: the gate's evidence is watchable.

### Decision 5 ‚Äî Failure halts, gates stay closed, partial state is named

Any step exiting nonzero throws `EnactmentError.stepFailed(index, exitStatus, stderr tail)` and ends the stream. Gated steps are unreleased by construction. No rollback in v1: rsync re-runs converge, tar partials are visible files the operator can see and act on. The error names the step so the panel can point at the exact command that failed.

### Decision 6 ‚Äî The fixture learns rsync and how to reach itself

The sshd container carries no rsync and there is one of it. Fixture surgery: `start` installs rsync in the container and provisions a self-referential ssh identity‚Äîthe fixture key and an in-container `~/.ssh/config` alias (`fixture-self` ‚Üí localhost:2222)‚Äîso a "cross-host" rsync from the container to `fixture-self` exercises the real command path, distinct directories standing in for distinct hosts. The tar proxy pipeline needs no surgery: its halves run from the operator's machine (the test process) toward the container, which is the real shape.

### Discovery (deferred to execution) ‚Äî progress2's exact fields and the pipeline's backpressure

The parser hardens against the fixture's rsync version live. The pipeline's write path must respect the ho-01 lesson‚Äîno blocking read starvation‚Äîwhile letting pipe backpressure do its job; the exact FileHandle discipline lands in execution against real streams.

## Phase 3 ‚Äî Reflect

**progress2 against the real rsync.** Bytes and percent held exactly as the synthetic battery assumed‚Äî`1,234,567  45%` shapes, comma-grouped, carriage-return refreshed‚Äîand the live bar finished at 1.0 on the nose. The recorded stream from the fixture's rsync 3.4.3 now rides the corpus, so a parser regression breaks against reality. One structural note: the parser buffers across chunk boundaries because a progress line split mid-number is routine, not exotic‚Äîthe first synthetic test for it caught the naive implementation before any fixture did.

**The pipeline under backpressure.** Nothing measurable at fixture scale‚Äîthe 100KB payload streamed through the counter without a visible dent. The design stands on the mechanism rather than the measurement: the counting pump blocks on a full consumer pipe, which pauses the producer read, which is the pipe's own flow control doing its job. The checksum comparison is the fidelity proof‚Äîbyte-identical after transplant. Worth re-measuring when a UI/UX session pushes real gigabytes.

**Verification shape.** Counts released the gates correctly in every live trace, and the mismatch battery proves a short count keeps the delete unrun. But counts are a thin truth‚Äî`find | wc -l` says how many, not how much. ho-06.2's zfs path verifies differently (the receive either lands whole or fails), and ho-08's panel may want size totals alongside counts. The `VerificationReport` shape can grow fields without breaking the gate contract; noted, not built.

**One naming correction, one CI honesty.** PlanStep grew a structured `Pipeline`‚Äîthe command string and the halves are composed together by the engine, so the enactment can spawn what the panel shows without re-parsing shell. And the CI runner ships openrsync, which the probe's own dotted-version rule classifies as no-usable-rsync‚Äîso the rsync-direct live test skips itself there by the same fact ho-06's ‚â•3.1 requirement reads. The tar proxy and every gate test run everywhere.

**Followups for ho-06.2.** The zfs transports throw `unsupportedTransport` typed‚Äî06.2 lifts it. The zfs step sequence is already composed and gated by ho-05; 06.2 enacts it against the pool VM through the same event stream, decides zfs-shaped verification (receive success + dataset existence beats file counts), and exercises the proxied zfs pipeline through the same `SSHPipeline` pump the tar path proved.
