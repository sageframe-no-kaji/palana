# ho-10.3 ‚Äî The ZFS Pane Mode

The answer to the two-cursor failure. In the ho-10.1 hands session the practitioner's destroy aimed at the panel's tree selection while his pane cursor sat on `angrybird` ‚Äî two cursors, two truths, one dangerous verb. His framing, ratified and sealed: **ZFS management is a separate activity and wants an explicit boundary.** This ho gives it one: a pane enters zfs mode, its ground shifts, its rows become the host's dataset tree, and every ZFS verb targets the one cursor the operator is looking at. The floating panel demotes to launcher and overview ‚Äî it stops being a mutation surface.

**Out of scope:** The `sudo -n` mount capability, the sudoers helper, and per-tool plugin toggles (ho-10.2, still sealed). Pool visualization beyond what the tree shows. Multi-host tree views. Editing properties beyond ho-10.1's vocabulary.

---

## Phase 1 ‚Äî Think

### Decision 1 ‚Äî Mode lives on the pane model; the core stays pure

A pane in zfs mode shows datasets, not files ‚Äî but `PaneState`/`PaneIntent` (core) encode file-listing truth and stay untouched. `PaneModel` (app) gains `paneMode: .files | .zfs`, and in zfs mode the pane's Table renders dataset rows from the cached `zfsTopology` fact (the ZFS panel's tree machinery, promoted: selection model, depth indent, mountpoint annotations, unmounted-dimmed-but-selectable). One cursor: the Table's own selection IS the target dataset. The file cursor and path are preserved under the mode and restored on exit.

### Decision 2 ‚Äî The boundary is loud: ground shift, header, footer

Entering zfs mode shifts the pane's ground toward the plugin hue (`Theme.plugin` low-opacity wash ‚Äî the burnt-umber family already speaks "plugin" on the strip), the header reads `zfs ¬∑ <host>`, and the footer names the exits. No mistaking which activity a pane is in from any distance ‚Äî that is the entire point.

### Decision 3 ‚Äî Enter and leave: `Z` on a pane, Esc comes home

`Z` (terminal-focus grammar, currently the panel toggle) now enters zfs mode on the focused pane when its host has the `.zfs` capability ‚Äî refused with the plain reason otherwise. Esc leaves (mode restores files, cursor, path). The floating panel keeps `‚áßZ` as its summon: it remains useful as a glance-overview and a launcher ("open in pane" gains "open as zfs mode"), but its verb rows go ‚Äî the panel no longer mutates. One mutation surface, one cursor.

### Decision 4 ‚Äî The verbs ride the existing gather unchanged

In zfs mode the ho-10.1 letters (`c x r n k b m i`) fire on the pane's cursor dataset through the same `runWorkbenchMutation(_:on:dataset:)` route, the same gathers, the same plan-then-Enter gate, the same pool-root refusal, the same typed destroy. Nothing in the mutation path changes ‚Äî only where the target comes from. ‚Üë‚Üì/jk walk the tree (the pane's normal cursor keys), Enter on a mounted dataset exits zfs mode INTO that dataset's mountpoint (the natural "go there"), right-click carries the verb menu.

### Decision 5 ‚Äî Post-run truth on the mode's own surface

A finished ZFS plan already refreshes panes and re-discovers facts (ho-10.1). A pane in zfs mode re-renders its tree from the fresh topology on the same signal ‚Äî the created dataset appears, the destroyed one goes, without leaving the mode.

  wash 0.12) so the mode is unmistakable ‚Äî answering "WTH is that?" from the hands.
