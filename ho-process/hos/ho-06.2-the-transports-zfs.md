# ho-06.2 — The Transports: zfs send/receive

Enactment, second half. Lifts `EnactmentError.unsupportedTransport`: the zfs step sequences ho-05 composed—snapshot, send piped to receive, gated cleanups and destroys—enact through the same event stream, against the throwaway pool, never a live host. The proxied zfs pipeline rides the same `SSHPipeline` pump the tar path proved.

**Out of scope:** any queue, any rollback. Mount management—pālana receives datasets, it does not run the host's mount policy.

**Resolves deferred decisions** (from the ho-overview):

- Progress parsing specifics—`zfs send -v` cadence (deferred decision 5, zfs half)

**Carries from ho-06.1:** the event stream, the gate machinery, the pipeline pump, and the note that zfs verification is not a file count.

---

## Phase 1 — Think

### Decision 1 — Verification grows a second shape: the dataset received

`find | wc -l` is the wrong truth for a block stream. The zfs stream is checksummed end to end by zfs itself—a receive that exits clean IS the byte verification—so the gate's question becomes existence: `zfs list -H -o name <child>` on the destination answers it. `VerificationReport` becomes an enum—`counts(source:destination:)` and `datasetReceived(name:exists:)`—with `matched` computed per case. The 06.1 battery updates mechanically; the gate contract does not move.

### Decision 2 — Two composition corrections, recorded here

ho-05's zfs commands meet reality twice. **`zfs receive` gains `-u`:** on Linux, mounting is root's alone regardless of `zfs allow` delegation, so a bare receive by a delegated user lands the dataset and then fails the auto-mount—an exit code that lies about the transfer. `-u` receives without mounting and the plan says so; mounting after is the host's mount policy, not pālana's. **`zfs send` gains `-v`:** progress for the forwarded path comes from send's own stderr cadence—a header line with the estimated size, then per-second byte lines. A `ZfsSendProgress` parser (sibling of the rsync one) reads bytes and computes fraction against the estimate. The proxied path already has the pipeline's byte counter.

### Decision 3 — The Linux delegation wall, named, not papered over

A move's final step destroys the source dataset, and destroying a *mounted* dataset needs an unmount—root's, on Linux. Enactment does not sudo and does not improvise: if the delegated user cannot unmount, the destroy fails typed and visible after a verified transfer—the copy landed, the source stayed, the panel shows exactly which command refused. That is the honest partial state. The fixture exercises the clean path with an unmounted source; the mounted-source reality is recorded for the field-use docs.

### Decision 4 — The pool plays two hosts, delegated

Fixture surgery in `zfs-fixture.sh`: a `zfs-self` alias operator-side (the lima config's Host line learns a second name) and inside the VM (self-keyed, localhost:22), plus `zfs allow` delegation on the pool for the VM user—send, snapshot, receive, create, destroy, hold, mount. Test datasets are throwaway children (`palana/t62-*`), created and torn down per case.

### Discovery (deferred to execution) — send -v's exact lines and delegation's real perimeter

The `-v` format and estimate line harden against the pool's zfs 2.4 live and get recorded. Whatever `zfs allow` genuinely cannot delegate on this Ubuntu lands in Reflect as the field truth.
