# ho-10.4 ‚Äî The Mount Seam

The root wall, answered where it can be answered. The ho-10.1 hands sessions mapped it in anger: on Linux, mounting is root's ‚Äî a delegated create lands unmounted, a delegated destroy of a mounted dataset dies on the unmount, and pƒÅlana had no mount verb at all, so an unmounted dataset was a dead end ("we still have serious mount issues"). This ho pulls the `sudo -n` capability forward from ho-10.2's sealed scope and gives the vocabulary its missing verbs: mount and unmount, composed with `sudo -n` when the host grants it, refused in plain words when it doesn't. The fixture VM's user has passwordless sudo, so the whole seam proves end-to-end against the throwaway pool; real hosts light up when the operator adds the sudoers line (ho-10.2's helper remains future work, with the settings trio).

**Out of scope:** the sudoers-helper and per-tool toggles (ho-10.2's remainder). Auto-mount after create (a natural follow-on; noted, not built). Any use of `sudo` beyond `zfs mount`/`zfs unmount` ‚Äî the capability is scoped to the two verbs, never a general escalation.

---

## Phase 1 ‚Äî Think

### Decision 1 ‚Äî The capability is a fact: `sudoNoPassword`, probed like any other

`HostFacts` gains `sudoNoPassword: Bool` (wrapped in the standard dated fact), gathered during `discover` with one cheap, harmless round trip: `sudo -n true` ‚Äî exit 0 means the host grants passwordless sudo, anything else means no. No password is ever sent or prompted; `-n` is the whole point. The fact renders in the field view like every other fact.

### Decision 2 ‚Äî Two verbs, honest keys: `m` mounts, `u` unmounts, mountpoint moves to `p`

`ZFSMutation` gains `.mount(dataset:)` and `.unmount(dataset:)`. Verb keys follow the muscle memory that exists: `m` = mount, `u` = unmount (umount, as every Unix hand types it), and set-mountpoint ‚Äî which held `m` ‚Äî moves to `p` (path/property). Field-less gathers both: the target is the cursor dataset, there is nothing to type, Enter shows the plan.

### Decision 3 ‚Äî `sudo -n` appears IN the composed command, never around it

compose renders `sudo -n zfs mount <ds>` / `sudo -n zfs unmount <ds>` ‚Äî the escalation is read in the plan like every other fact of the command. Verify steps stay unprivileged (`zfs list -H -o name,mounted`). `PlanStep.Role` reuses `.property` for both (they change where a dataset is reachable, not what exists ‚Äî same family as mountpoint).

### Decision 4 ‚Äî Availability gates on BOTH facts, and the refusal names the wall

The verbs require `.zfs` AND `sudoNoPassword == true`. A ZFS host without the sudo grant shows the verbs dimmed with the plain reason: "mounting needs root on Linux ‚Äî grant passwordless sudo for zfs, or use the shell". No composing toward a guaranteed failure.

### Decision 5 ‚Äî Fixture-proven, agent-hands-off

Unit tests pin the compose strings, the validation, and the capability gating. The live round trip (mount an unmounted dataset, see it in the tree and the pane, unmount it back) belongs to the practitioner's hands against the fixture pool ‚Äî the building agent runs unit tests only and never touches the running VM.
