# ho-10.4 — The Mount Seam

The root wall, answered where it can be answered. The ho-10.1 hands sessions mapped it in anger: on Linux, mounting is root's — a delegated create lands unmounted, a delegated destroy of a mounted dataset dies on the unmount, and pālana had no mount verb at all, so an unmounted dataset was a dead end ("we still have serious mount issues"). This ho pulls the `sudo -n` capability forward from ho-10.2's sealed scope and gives the vocabulary its missing verbs: mount and unmount, composed with `sudo -n` when the host grants it, refused in plain words when it doesn't. The fixture VM's user has passwordless sudo, so the whole seam proves end-to-end against the throwaway pool; real hosts light up when the operator adds the sudoers line (ho-10.2's helper remains future work, with the settings trio).

**Out of scope:** the sudoers-helper and per-tool toggles (ho-10.2's remainder). Auto-mount after create (a natural follow-on; noted, not built). Any use of `sudo` beyond `zfs mount`/`zfs unmount` — the capability is scoped to the two verbs, never a general escalation.

---

## Phase 1 — Think

### Decision 1 — The capability is a fact: `sudoNoPassword`, probed like any other

`HostFacts` gains `sudoNoPassword: Bool` (wrapped in the standard dated fact), gathered during `discover` with one cheap, harmless round trip: `sudo -n true` — exit 0 means the host grants passwordless sudo, anything else means no. No password is ever sent or prompted; `-n` is the whole point. The fact renders in the field view like every other fact.

### Decision 2 — Two verbs, honest keys: `m` mounts, `u` unmounts, mountpoint moves to `p`

`ZFSMutation` gains `.mount(dataset:)` and `.unmount(dataset:)`. Verb keys follow the muscle memory that exists: `m` = mount, `u` = unmount (umount, as every Unix hand types it), and set-mountpoint — which held `m` — moves to `p` (path/property). Field-less gathers both: the target is the cursor dataset, there is nothing to type, Enter shows the plan.

### Decision 3 — `sudo -n` appears IN the composed command, never around it

compose renders `sudo -n zfs mount <ds>` / `sudo -n zfs unmount <ds>` — the escalation is read in the plan like every other fact of the command. Verify steps stay unprivileged (`zfs list -H -o name,mounted`). `PlanStep.Role` reuses `.property` for both (they change where a dataset is reachable, not what exists — same family as mountpoint).

### Decision 4 — Availability gates on BOTH facts, and the refusal names the wall

The verbs require `.zfs` AND `sudoNoPassword == true`. A ZFS host without the sudo grant shows the verbs dimmed with the plain reason: "mounting needs root on Linux — grant passwordless sudo for zfs, or use the shell". No composing toward a guaranteed failure.

### Decision 5 — Fixture-proven, agent-hands-off

Unit tests pin the compose strings, the validation, and the capability gating. The live round trip (mount an unmounted dataset, see it in the tree and the pane, unmount it back) belongs to the practitioner's hands against the fixture pool — the building agent runs unit tests only and never touches the running VM.

`HostFacts.sudoNoPassword` + the discover probe; `.mount`/`.unmount` in `ZFSMutation` with composes and validation; the two `WorkbenchVerb`s with the key reshuffle (`m`/`u`/`p`); availability gating with the plain refusal; the unit battery. → the ho doc is the spec.

## Phase 3 — Reflect

The seam grew past its single AT across a 2026-07-15 hands session — three more
ATs, each on his live verdicts:

- **AT-01 (`b0645ba`)** — the fact and the verbs. `HostFacts.sudoNoPassword`, its
  discover probe (evolved in-session to `sudo -n true || sudo -n -l zfs mount` —
  blanket OR scoped grant, his question found the gap); `.mount`/`.unmount` with
  `sudo -n zfs mount/unmount` read IN the plan; keys `m`/`u`, set-mountpoint → `p`;
  `CapabilityRequirement` gating on BOTH facts with the plain refusal. The nil
  fact reads "not asked yet", never the root-wall sentence.
- **AT-02 (`b49e42f`)** — the implicit-unmount heal. destroy / set-mountpoint /
  clear-mountpoint on a MOUNTED dataset weave `sudo -n zfs unmount` ahead (and a
  trailing `sudo -n zfs mount` for the two mountpoint verbs), so zfs's own
  root-only implicit unmount never blocks the delegated user. `MutationInput` +
  `PlanRequest.targetMounted` carry the mounted fact from the surface to the pure
  composer. Also **F1**: the field view grew the `sudo` chip to match the host map.
  His decision, live: "1 step would be better… its clear about what it is going to
  do in the pane."
- **AT-03 (`dc6abf7`)** — the recursive toggle got a key. `space` flips
  `zfsRecursive` in both the field-less (destroy) and text (snapshot, rollback)
  gathers — space over `r` because a typed name can hold `r` but never a space.
  It was mouse-only before; the hands session sailed past it every time.
- **AT-04 (`869763d`)** — the property change stops trying to mount.
  set-mountpoint composes `zfs set -u mountpoint=…` (`-u` = set without the
  auto-mount that threw a mid-plan "insufficient privileges" warning). `zfs inherit`
  has no `-u` (only `-rS` on 2.4.1), so clear-mountpoint keeps its cosmetic warning
  — the explicit `sudo -n zfs mount` still lands it. `-u` needs OpenZFS 2.2+
  (every host in play); a version-gate fallback is deferred.

**His hands verdicts (2026-07-15, fixture pool):** the heal fired correctly on a
pane-driven destroy of a mounted dataset (`palana/armedguard` — unmount then
destroy, one step, "It worked"); the F1 sudo chip shows in the field; space flips
recursive ("SMOOP"); the `-u` set-mountpoint runs clean, no warning. Clarified and
accepted as correct: clear-mountpoint inherits the parent's path
(`baserow → /opt/services/baserow`); set-mountpoint on an unmounted dataset stays
unmounted — the heal preserves mount state, and `m` mounts it if wanted.

**Deferred / banked (named, not built):** recursive-destroy of a subtree whose
CHILDREN are separately mounted (single `unmount` covers only the named dataset);
set-mountpoint auto-mounting a previously-unmounted dataset (a preference, his
call, a fresh ho if wanted); the `-u` version-gate for pre-2.2 hosts; the sudo
fixture `zfs-pool.json` re-capture (predates the sudo probe — nothing asserts on
it). The sudo-explainer (sudoers line at the refusal + settings reference) is the
last live piece of ho-10.2's old scope — its own fresh ho.
