# ho-07.5 — The BusyBox Userland

Inserted when zencat answered: the probe classified its BusyBox as BSD, the BSD listing command asked for a `stat` binary the box does not carry, and usage spew surfaced as a typed read failure. The practitioner's firmware box deserves better than a riddle. This ho gives the probe a third honest answer and the Listing a path that speaks what BusyBox actually has.

The recon facts, read live from zencat (BusyBox v1.25.1, vendor-trimmed 2025): no `stat` binary at all. `find` carries only `-name`, `-mtime`, `-print0`, `-exec` — no `-mindepth`, `-maxdepth`, `-type`. `sh` lacks the `command` builtin. It HAS `awk`, and `ls -lane` works.

**Out of scope:** rsync/zfs anything on BusyBox hosts — the probe records their absence and ho-08's fact gates already route to tar. Recursive size facts on BusyBox — no walkable `find`, so plans show the inode floor with `totalSizeComplete == false`, the honesty ho-06.5 built. Mutating verbs beyond what the engine already composes (BusyBox has `mv`, `cp`, `rm`, `tar`).

---

## Phase 1 — Think

### Decision 1 — A third flavor, told by the busybox binary itself

`UserlandFlavor` grows `.busybox`. The probe's tell, in order: `stat --version` answers → GNU; else `busybox true` exits 0 → BusyBox; else BSD. A GNU host with busybox installed stays GNU (first tell wins); a Mac has no busybox and stays BSD. Old cached facts stay decodable — the new case is additive.

### Decision 2 — The listing is `ls -lane`, degradation named, never guessed

The choice was degraded fidelity versus refusal, and a browseable firmware box beats a locked one — but every degradation is named, in the parser's contract and the field docs: owner and group are numeric (`-n`), the mtime is `ls`'s own clock text with second precision at best and the remote's timezone unstated, and a name that breaks the line structure (a newline) refuses the whole listing loudly — `malformedListing`, never a silently wrong row. Kinds come from the mode string's first column; symlink targets from the ` -> ` split. `.` and `..` drop.

### Decision 3 — What BusyBox cannot do stays visible, not special-cased

`treeSizes` on `.busybox` returns no facts — no `-type f`, no walk — and the existing floor machinery does the honest work: plans over BusyBox directories show the inode floor and say it is one. The transports need nothing: ho-08's rsync gate already falls to tar when rsync is absent, and BusyBox tar unpacks a stream fine.

### Decision 4 — The stale-fact edge is ho-09's, named here

zencat's cache may still say BSD from the misclassified visit. The cache is deletable memory by design; the explicit re-probe control is the field view's (ho-09). Until then: delete `field-cache.json` or point at a fresh host name.

### Discovery (deferred to execution) — the fixture's userland

Whether a plain-Alpine sshd container (BusyBox userland, no coreutils) stands up cheaply beside the existing fixture. If it does, integration runs live; if not, the recon shapes and synthetic battery carry, and the fixture rides a follow-up.

## Phase 3 — Reflect

**The recon was true and insufficient — BusyBox flag sets are vendor-build-dependent.** zencat's 1.25 speaks `ls -lane`; Alpine's 1.37 refuses `-e` and offers `--full-time` instead, with a real timezone, which is better truth than the recon's rung. Decision 2's single command became a one-round-trip precision ladder: `-e` full seconds, `--full-time` full seconds with timezone, short dates as the floor — the parser owns all three shapes and still refuses any line it cannot, loudly.

**The probe tell held against the real thing.** A bare Alpine container answered `palana:flavor:BusyBox` on the first try — `busybox true` after GNU stat declines, exactly the ordered tell. The dedicated sshd-on-Alpine fixture was not built: the userland truths this ho needed (probe classification, ls shapes) came straight from `docker run`, and the over-the-wire BusyBox path rides the practitioner's own zencat drive. If CI wants the wire variant later, it is one fixture script away.

**Recapture ripples, all honest.** The probe command changed, so both recorded corpora re-recorded against live fixtures — and the container corpus now truthfully shows the rsync ho-06.1 installed, which the old assertion predated. The rebuilt pool VM's self-hop key had gone stale; `ssh-keygen -R` inside the VM, per its own error text.

**Carried:** zencat's field cache may still remember the BSD misclassification — deleting `field-cache.json` is the interim answer, the explicit re-probe control is ho-09's.
