# Ho-10.1-AT-03 — The surface: the mutation path and the gather

**Goal**

Wire the `.mutation` route through the app: a ZFS verb in the strip → the target dataset resolved from where the focused pane stands → a parameter gather in the panel → `planRequest` → `PlanEngine.plan` → the plan renders → **Enter enacts, nothing else does**. The `ZFSMutationTool` (AT-02) joins the Workbench beside the reads tool. A hands session settles the feel afterward — build the committed starting shape, not speculative chrome.

**The law this task exists to honor (Decision 4):** a ZFS mutation NEVER auto-enacts on gather submit. rename/create skip the second Enter because the name field is the confirmation; a `zfs destroy -r` is confirmed by *reading the composed command*. Gather submit composes and renders the plan at `.ready`; the existing Enter-at-`.ready` path enacts. Do not special-case Enter for ZFS — the generic gate already does this.

**Required Changes**

1. **The tool joins the session.** `PalanaSession` gains `let zfsTool = ZFSMutationTool()` beside `readsTool`. The terminal-focus hint dispatch in `handle()` searches `readsTool.verbs + zfsTool.verbs` (key hints are disjoint by construction; keep first-match).

2. **The strip grows a zfs section.** Below the read chips: a divider, a small section caption (`zfs`, `Theme.inkFaint`, the section is visual only), then one chip per `zfsTool.verbs` — same `StripChip`, same availability refresh (loop over both tools' verbs), same local-honesty guard (the existing `.zfs`-on-this-Mac branch already covers them). Disabled state and tooltips work exactly as reads do.

3. **`runWorkbenchVerb` routes by kind.** `.read` keeps today's path untouched. `.mutation`:
   - Guards mirror the read path (terminal not busy, host present, local-honesty, availability checked async).
   - Resolve the target dataset: `field.facts(for: host)?.zfsTopology?.value` then `ZFSTopology.datasetContaining(focusedPane.state.path, in: topology)`. No containing dataset → `operation.appendToolError("no dataset holds \(path) on \(host)")` and stop. (Availability already guarantees topology exists; the pane may still stand outside any dataset.)
   - Hand off: `operation.beginZFSMutation(verb, tool: zfsTool, host: host, dataset: dataset.name)`.

4. **The gather (`OperationModel+ZFS.swift`).** `beginZFSMutation(_:tool:host:dataset:)` follows `beginNaming`'s phase law verbatim (enacting → just show panel; gathering → cancel; naming → reset; then fresh). State: pending verb/tool/host/dataset plus `zfsRecursive: Bool` (reset to false each begin, cleared in `reset()`).
   - **Verbs that gather** (`verb.gather` has `needsText` or `offersRecursive`): enter the `.naming` phase so the key monitor stands down and the panel's field row appears. The label is composed here with the real names — plain sentences, e.g. create: `"name the new dataset — a child of tank/data  (⏎ shows the plan)"`; destroy: `"destroy tank/data — ⏎ shows the plan, nothing runs yet"`. Rename prefills the field with the full current dataset name, selected (ho-9.1's prefill idiom). Non-text verbs (destroy) show the row without a field.
   - **Verbs that gather nothing** (`zfs-clear-mountpoint`): skip the field entirely — compose immediately (step 5) and land at `.ready`.
   - `commitNaming` routes: when a pending ZFS verb exists, delegate to `commitZFSGather(text)` instead of the rename/create path. Empty required text dismisses quietly (`reset()`), matching `commitNaming`'s posture. A rename submitted with the name unchanged (the prefill unedited) also dismisses quietly — ho-9.1's posture; do not let it reach the engine's `zfsRenameNamesIdentical` refusal.
   - **The field-less gather row** (destroy): while it is live, Return commits, Esc dismisses, and the recursive toggle is clickable; typed keys must not leak to the panes. Choose the seam that already exists (`isNaming` stands the monitor down; a focusable row or hidden-focus mechanism is fine) — the law is behavior, the mechanism is yours. If the monitor must learn one new branch, keep it inside the existing naming guard.
   - **The recursive toggle** renders in the field row when `verb.gather?.offersRecursive == true` — a small labeled Toggle (`recursive — includes everything beneath`), binding to `zfsRecursive`. It must be reachable by mouse at minimum; keyboard reach is a feel question for the hands session.

5. **Compose and render.** `commitZFSGather` builds `MutationInput(target: dataset, text: trimmed, recursive: zfsRecursive)`, calls `tool.planRequest(for:on:input:)`, then `PlanEngine.plan(request, facts: PlanFacts())`. Success → `plan` set, `phase = .ready` — **and stop; no `enact()` call** (Decision 4). A nil `planRequest` dismisses quietly (malformed gather = non-event); a `PlanError` renders as a failure in the panel (`Self.describe`, existing idiom).

6. **The panel renders a ZFS plan honestly.** In `planBlock`, the size line (`"0 entries · Zero bytes"`) stands down for `plan.operation == .zfs` — it would be noise beside a dataset mutation. Everything else stands: the header (`zfs · zfs mutation`), the route line (`host:dataset` — the locus carries the dataset from AT-02), the transport line (`runs locally on the one host involved`), the steps with their visible `.verify` reads. Do not invent a new block.

7. **After a finished ZFS run, the world re-reads.** On `.finished` for a `.zfs` plan: refresh any pane standing on that host (mountpoint moves and destroys change what listings show), and kick one field re-discovery of the host (`engine.field.discover`, fire-and-forget Task) so the topology fact — dataset names, mountpoints, the `◆` markers, future verb targeting — carries the new truth. Find the existing `.finished` handling in `handle(_ event:)` / the finish path and add the ZFS branch there; keep it non-blocking.
