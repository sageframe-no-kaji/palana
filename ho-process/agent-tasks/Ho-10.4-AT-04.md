# Ho-10.4-AT-04 ‚Äî The property change stops trying to mount

## Why

The hands session found a noisy edge in the AT-02 heal. `zfs set mountpoint=‚Ä¶`
and `zfs inherit mountpoint` each attempt their OWN auto-remount of the dataset
(canmount=on), and that mount is root-only for the delegated user ‚Äî so a
`cannot mount '<ds>': Insufficient privileges` warning fires mid-plan even when
the overall operation succeeds:

- **set-mountpoint on an unmounted dataset**: the heal adds no unmount/mount
  (correct ‚Äî it was unmounted), but `zfs set mountpoint` still tries to auto-mount
  and warns; the property is set regardless.
- **set/clear-mountpoint on a mounted dataset**: our explicit `sudo -n zfs mount`
  lands it correctly, but the property command's own auto-mount warns first.

## Decision (made)

Use **`zfs set -u mountpoint=<path>`** for `.setMountpoint` ‚Äî `-u` sets the
property WITHOUT mounting/remounting (confirmed on the fixture, zfs-2.4.1). Then
the explicit unmount/mount steps (mounted case only) do all the mounting, and the
unmounted case simply sets the property and stays unmounted ‚Äî no warning, no
failed mount.

Ship `-u` **unconditionally** (option a). It requires OpenZFS 2.2+ (Oct 2023),
which covers every host in play. A version-probe fallback for older hosts is
explicitly deferred ‚Äî note it in the Reflect, do not build it.

**`zfs inherit` has no `-u`** (only `-rS` on 2.4.1). So `.clearMountpoint` keeps
its current compose; its mid-plan auto-mount warning is cosmetic (the explicit
`sudo -n zfs mount` still lands it at the inherited path). Name this in the
Reflect as a known-cosmetic wart, not a bug ‚Äî do NOT try to suppress stderr or
hack around it.

## Scope

- `Sources/PalanaCore/Plan/PlanEngine+ZFSComposition.swift`,
