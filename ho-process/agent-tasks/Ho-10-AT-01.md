# Ho-10-AT-01 — The Workbench protocol and the read-only tool

**Goal**

The Workbench API kamae-2 named, built in PalanaCore and proven by one read-only tool. A `WorkbenchTool` protocol whose verbs carry a capability requirement and a kind, a coordinator that gates a verb against the Field's cached facts and runs a read verb's composed command through the Conduit, and the first tool declaring `df -h`, `zfs list`, `zpool status`, `zpool list`. The mutation seam is named in the types and honored by the routing—not built. No Surface code.

**Context**

ho-10 Decision 5: the Workbench protocol is small. A tool declares verbs; on registration it holds the Conduit and the Field (kamae-2's two capabilities), and never sees PalanaCore internals. A verb's *kind* is the seam—a **read** verb composes a command the coordinator runs through the Conduit, streaming raw bytes with no parsing (transparency is the feature); a **mutation** verb composes a `PlanRequest` for the Plan Engine to classify and the operator to arm with Enter (kamae-2: only the Plan Engine classifies, the Surface never composes a mutating command). v1 builds the read path only. The mutation path is named in the types and exercised by the routing distinction, so the later ZFS tool declares mutation verbs without new machinery or a core change.

Decision 1: a verb aims at a host the caller names (the app passes the focused pane's host). Decision 2: gating keys on the Field's cached facts, never a fresh probe—`df` stays available unless the host is *known* unreachable (its own raw output is its reachability test); the `zfs` verbs are available only where a `zfsTopology` fact is present (probed and zfs-bearing), and read as "not yet probed" where no facts exist versus "no zfs here" where facts exist without zfs. No silent absence—an unmet verb carries a plain reason.

Read before writing: `Sources/PalanaCore/Conduit/Conduit.swift` (`run(on:_:) -> RunningCommand`, `RunningCommand.collect() -> CommandResult`, `CommandResult.stdoutText`/`exitStatus`), `Sources/PalanaCore/Conduit/RecordedConduit.swift` (the test corpus shape—`ConduitTranscript`, `Entry(host:command:stdout:stderr:exit:)`), `Sources/PalanaCore/Field/HostFacts.swift` (`reachability`/`capability`/`zfsTopology`, `Dated`, `Reachability`), `Sources/PalanaCore/Field/Field.swift` (`facts(for:)`), `Sources/PalanaCore/Field/MountTable.swift` (the command-composition + pure-function precedent, and how a core type is tested against recorded conduits), `Sources/PalanaCore/Plan/*` for `PlanRequest`'s shape (referenced by the seam, not constructed here).

**Required Changes**

1. **`VerbKind`** — `public enum VerbKind: Sendable { case read, mutation }`.

2. **`CapabilityRequirement`** — `public enum CapabilityRequirement: Sendable { case reachable, zfs }`. A pure evaluator returns availability, not a bool, so the reason travels:
   - `func evaluate(host:facts:) -> VerbAvailability`
   - `.reachable`: `.available` unless `facts?.reachability?.value` is explicitly `.unreachable` → `.unmet("<host> is unreachable")`. Unprobed (nil facts) stays `.available`—the run is its own reachability test.
   - `.zfs`: `.available` when `facts?.zfsTopology != nil`; `.unmet("<host> has no zfs")` when facts exist without it; `.unmet("<host> not yet probed—probe from the field or map")` when facts are nil.

3. **`VerbAvailability`** — `public enum VerbAvailability: Sendable, Equatable { case available; case unmet(String) }`. The `String` is a plain sentence for the disabled button's reach.

4. **`WorkbenchVerb`** — `public struct WorkbenchVerb: Sendable { public let id: String; public let label: String; public let keyHint: String; public let requirement: CapabilityRequirement; public let kind: VerbKind }`.

5. **`WorkbenchTool`** — the protocol:
   ```swift
   public protocol WorkbenchTool: Sendable {
       var id: String { get }
       var label: String { get }
       var verbs: [WorkbenchVerb] { get }
       /// A read verb's composed command for a host. The tool composes; the
       /// Workbench runs. Never called for a mutation verb.
       func command(for verb: WorkbenchVerb, on host: String) -> String
   }
   ```
   The mutation seam is named as a protocol extension point, not built: a documented default `func planRequest(for verb: WorkbenchVerb, on host: String) -> PlanRequest?` returning `nil`, with a DocC note that mutation tools override it and the Workbench routes a `.mutation` verb through the Plan Engine—the ZFS tool's future ho fills this. Do not construct a real `PlanRequest`.

6. **`Workbench`** — the coordinator, `Sendable`, holding the Conduit and the Field handed in at init:
   - `func availability(of verb: WorkbenchVerb, on host: String) -> VerbAvailability` — reads `field.facts(for: host)`, delegates to `verb.requirement.evaluate`. Local host is reachable by construction (no facts needed for `.reachable`).
   - `func run(_ verb: WorkbenchVerb, of tool: WorkbenchTool, on host: String) async throws -> RunningCommand` — precondition `verb.kind == .read` (a `.mutation` verb here is a programmer error routed elsewhere—throw a typed `WorkbenchError.notARead` or assert per the codebase's guard style); compose `tool.command(for:on:)`, return `conduit.run(on: host, command)`. The caller collects or streams; this returns the live handle. No parsing, no exit-status interpretation—raw output is the contract.

7. **`SystemReadsTool`** — `WorkbenchTool`, `id "reads"`, `label "system reads"`, four verbs, all `.read`:
   - `df` — label "df", keyHint per the app's convention (leave a sensible letter, the app owns final hints), requirement `.reachable`, command `df -h`
   - `zfs-list` — label "zfs list", requirement `.zfs`, command `zfs list`
   - `zpool-status` — label "zpool status", requirement `.zfs`, command `zpool status`
   - `zpool-list` — label "zpool list", requirement `.zfs`, command `zpool list`

   `command(for:on:)` returns the literal command—the same string per host in v1 (the tool composes it; the host is a parameter the future may use). No flags beyond `-h` on df.

8. **Tests** — `WorkbenchTests`:
   - `availability` across fixtures: a reachable host (df available, zfs unmet "no zfs"), a zfs-bearing host (all available), an unreachable host (df unmet), an unprobed host (df available, zfs unmet "not yet probed"), local (df available).
   - the composed command strings for all four verbs.
   - the read run against a `RecordedConduit` corpus: a recorded `zpool status` and `df -h` return their raw stdout through `run(...).collect().stdoutText` unchanged.
   - the routing guard: `run` on a `.mutation` verb throws/asserts; `planRequest` default returns nil.

**Do Not**

- No Surface code—this task is entirely PalanaCore and its tests. The strip, the focus, the transcript are AT-02.
- Do not build the mutation path. `VerbKind.mutation` exists, `planRequest` is a named nil-returning seam, and `run` refuses a mutation verb. Nothing composes or enacts a mutating command.
- Do not parse, classify, or interpret read output. Raw bytes to the caller. The transparency is the whole point.
- Do not open or modify other PalanaCore types. The Workbench holds the Conduit and the Field through their existing public surface. If something needed is not public, stop and surface rather than widening core.
- Do not add a new SwiftPM target unless the build forces it—a `Workbench/` group inside PalanaCore satisfies kamae-2's "compiled-in." (Execution call per the ho's Discovery note.)

**Stop Condition**

If `PlanRequest` cannot be named by the seam without constructing one (a type-visibility problem), or if gating needs a fact the Field does not expose, stop and surface rather than reaching into core internals or fabricating a fact.
