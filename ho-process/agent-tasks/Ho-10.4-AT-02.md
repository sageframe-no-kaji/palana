# Ho-10.4-AT-02 — The implicit-unmount heal

## Why

The mount seam (AT-01) sudo-wrapped the two verbs it named — `mount`/`unmount`.
But `zfs destroy` and `zfs set mountpoint`/`zfs inherit mountpoint` on a
**mounted** dataset trigger an *implicit* unmount to do their work, and on Linux
that unmount is root-only. As the delegated (non-root) user those commands fail
with `cannot unmount '<path>': permission denied` (exit 1 / 255). The hands
session hit this live on `palana/pjb`. As root these commands unmount silently;
the delegated user must clear the mount first.

Practitioner's decision (ratified live): **one step is better** — pālana should
weave the `sudo -n zfs unmount` in as a first plan step, because the operator
reads the whole sequence in the pane before Enter, so nothing is hidden.

## Scope

When the target dataset is **currently mounted**, the composed plan becomes:

- **destroy (mounted)** → `sudo -n zfs unmount <ds>` · `zfs destroy[ -r] <ds>`
- **set-mountpoint (mounted)** → `sudo -n zfs unmount <ds>` · `zfs set mountpoint=<path> <ds>` · `sudo -n zfs mount <ds>`
- **clear-mountpoint (mounted)** → `sudo -n zfs unmount <ds>` · `zfs inherit mountpoint <ds>` · `sudo -n zfs mount <ds>`

When the target is **unmounted**, compose is unchanged (no unmount, no remount —
today's behavior, which already works).

The trailing `sudo -n zfs mount` on set/clear-mountpoint restores the dataset to
mounted at its new path, matching what root does automatically. Destroy has no
remount — the dataset is gone.

**Out of scope (name in the Reflect, do not build):** recursive destroy (`-r`)
of a subtree whose *children* are separately mounted — a single
`sudo -n zfs unmount <ds>` unmounts only the named dataset. Handle the
single-dataset case now; flag the mounted-children case as a follow-up.

## The seam (already traced — build to this)

Compose is a **pure function** (`PlanEngine.composeZFSMutation(request:)`, no
Field/no I/O), so the mounted fact must ride in on the request. The surface
already knows it (`dataset.mounted`).

1. **`MutationInput`** (`Sources/PalanaCore/Workbench/WorkbenchTool.swift`) —
   add `mounted: Bool = false` (surface→tool carrier; keep the defaulted init
   back-compatible).
2. **Surface call sites** — wherever `MutationInput(...)` is constructed for the
   ZFS verbs (trace from `Sources/Palana/PalanaSession+ZFS.swift` and
   `Sources/Palana/PaneModel+ZFSMode.swift` — both already read
   `dataset.mounted` nearby), pass `mounted: dataset.mounted`.
3. **`PlanRequest`** (`Sources/PalanaCore/Plan/PlanRequest.swift`) — add
   `public var targetMounted: Bool = false` (consonant with the existing
   optional ZFS context fields), threaded through its memberwise init.
4. **`ZFSMutationTool.planRequest(for:on:input:)`**
   (`Sources/PalanaCore/Workbench/ZFSMutationTool.swift`) — set
   `targetMounted: input.mounted` on the `PlanRequest` it returns.
5. **`PlanEngine+ZFSComposition.swift`** — `composeZFSMutation(request:)` reads
   `request.targetMounted` and passes it to the two sub-composers
   (`composeZFSDatasetMutation`, `composeZFSSnapshotAndPropertyMutation` — add a
   `targetMounted: Bool` parameter). Branch the `.destroyDataset`,
   `.setMountpoint`, `.clearMountpoint` cases per the Scope table above. Reuse
   `ShellQuote.quote`. The unmount/mount steps use `role: .property` (same as the
   AT-01 mount/unmount steps); the existing verify steps stay unprivileged and
   unchanged.
